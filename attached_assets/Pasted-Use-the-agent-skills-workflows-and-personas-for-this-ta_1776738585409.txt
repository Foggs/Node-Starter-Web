## Task #25: Move session-readiness check into shared middleware

## Problem

checkSessionReady is currently defined inside
artifacts/api-server/src/routes/coaching.ts. As more routes
need the same gate, each either duplicates the logic or imports
it from an unrelated file. This creates drift risk — if the
readiness logic needs to change (e.g. adding sub_type to the
required fields), it has to be found and updated in multiple
places.

## Incremental implementation plan (follow this order)

**Slice 1 — Audit current usage**
- Find every location in the codebase that calls or duplicates
  checkSessionReady or an equivalent session gate check
- List each file, line number, and what fields it checks
- Do not change any code in this slice
- Document findings as a comment block at the top of the
  new middleware file before writing any logic

**Slice 2 — Create the middleware file**

Create: artifacts/api-server/src/middlewares/sessionReady.ts

The middleware must:
- Be a standard Express RequestHandler signature:
  (req, res, next) => void
- Check fields in this exact order:
    1. Session exists
    2. consent_given === true
    3. scenario is set and is a known enum value
    4. persona is set and is a known enum value
    5. voice_id is set OR voice_cloned === false
- On any failure: return res.status(401).json({
    error: 'Session not ready',
    missing: '<first missing field name>'
  })
- On success: call next()
- Export as named export: checkSessionReady
- Export the required fields list as a separate named export:
  SESSION_REQUIRED_FIELDS
  so callers can introspect what the gate requires without
  reading the implementation

**Slice 3 — Update coaching.ts**
- Remove the local checkSessionReady definition
- Import checkSessionReady from
  ../middlewares/sessionReady
- Wire it to all routes that previously used the local version
- Verify the file has no remaining local session-gate logic

**Slice 4 — Audit all other gated routes**
- For every other route file identified in Slice 1 that
  duplicates or partially duplicates the gate:
  - Remove the local logic
  - Import and apply the shared middleware
- Do not change any route logic beyond the middleware wiring

**Slice 5 — Run the full test suite**
- Re-run all 8 tests from the Prove-It suite
- All 8 must pass before proceeding to review
- If any test fails, fix before proceeding — do not skip

## Activate: code-reviewer persona

You are a Staff Engineer. Apply five-axis review to the
middleware and all updated route files:

1. Correctness — Does the middleware enforce the exact field
   chain in the right order? Does it correctly handle the
   voice_cloned: false fallback as a valid terminal state?
   Does it return the right status code (401 not 403)?
   Is the missing field name surfaced in the error response?

2. Readability — Is the middleware self-documenting?
   Could another engineer understand what "session ready"
   means without reading the onboarding flow? Is
   SESSION_REQUIRED_FIELDS useful and accurate?

3. Architecture — Is there a single source of truth for
   the readiness definition? If sub_type is added to
   required session fields in future, is there exactly
   one place to update? Is the middleware in the right
   directory per the project's existing conventions?

4. Security — Is the middleware applied to every route
   that was previously gated? Are there any routes that
   handle sensitive operations (voice, coaching, replay,
   export) that are now ungated by mistake? Cross-check
   against the API endpoint list in the SDD.

5. Edge cases — What happens if req.session exists but
   is malformed (e.g. fields present but wrong type)?
   What if voice_id is an empty string rather than
   undefined? What if scenario is a valid string but
   not a known enum value?
